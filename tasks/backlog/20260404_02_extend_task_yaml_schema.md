# Extend task YAML schema with phase and dependency fields

## Problem

Task YAML has only `status` (and optionally `type`). Task ordering lives as prose in `implementation-plan.md`. Implementation skills cannot mechanically determine execution order — they rely on the agent re-reading prose and inferring. This is fragile, non-verifiable, and blocks future parallel execution.

## Scope

**In scope:**
- Define extended YAML header fields: `phase` (integer), `depends_on` (list of task identifiers)
- Update `rules/task-management.md` to document the new fields (required for plan-generated tasks, optional for ad-hoc tasks)
- Update `docs/spec-to-code-specification.md` to reflect the schema change

**Out of scope:**
- Modifying existing skills to use the new fields (that happens in tasks 06 and 09)
- Building dependency graph validation tooling (deferred to hooks in task 03 or MCP in v2)
- Changing task naming convention

## Acceptance Criteria

- AC1: `rules/task-management.md` documents `phase` and `depends_on` fields with semantics, data types, and examples
- AC2: Fields are required for tasks generated by `/stc:plan`, optional for tasks created via `/stc:create-task`
- AC3: `depends_on` references task filenames (without path) for unambiguous identification
- AC4: Spec document updated to reflect the schema

## Quality Gates

- Task management standard is internally consistent (no contradictions with existing fields or lifecycle rules)
- Example task in the standard demonstrates all fields
